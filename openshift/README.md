# Deploying Autoscale Arena to OpenShift

These steps deploy the app to a Red Hat OpenShift project (Developer Sandbox works) using the built-in Node.js S2I builder. You do not need cluster-admin. Run everything from the `openshift/` directory unless otherwise noted.

## Prerequisites

- `oc` logged in and a project selected: `oc project <your-project>`
- The repo pushed to a public Git URL (S2I pulls source from Git, not from your laptop)
- `package-lock.json` committed — S2I refuses to build without it

## One-time setup

```bash
# RBAC: ServiceAccount the frontend uses to watch pods
oc apply -f rbac.yaml

# Build the image from Git via S2I
oc new-app nodejs~https://github.com/<user>/autoscale-arena.git \
    --name=autoscale-arena-frontend

# Scale the auto-generated frontend Deployment to zero while we wait for the
# build. oc new-app boots a pod without our ServiceAccount or env vars, so it
# will fail readiness probes until `frontend.yaml` is applied below. Scaling
# to zero avoids the transient crash-loop and the scary logs that come with
# it. You'll scale it back up in the next section.
oc scale deploy/autoscale-arena-frontend --replicas=0

# Wait for the first build to finish
oc logs -f bc/autoscale-arena-frontend
```

`oc new-app` creates a Deployment, Service, ImageStream, and BuildConfig. We replace the Deployment and Service with the versions in `frontend.yaml` below so the ServiceAccount and env vars are wired correctly.

## Wire the frontend

```bash
# Replace the autogenerated Deployment + Service + Route with our version
# (frontend.yaml sets replicas: 1, so this also scales the pod back up)
oc apply -f frontend.yaml

# Check the pod rolls out cleanly
oc rollout status deploy/autoscale-arena-frontend
```

## Wire the worker (same image, different env)

```bash
# Reuse the ImageStream S2I just built
oc apply -f worker.yaml

# Point the worker Deployment at the ImageStream tag so rebuilds propagate.
# (frontend.yaml/worker.yaml use the `autoscale-arena-frontend:latest` image
# tag; trigger annotations tie the Deployment to the ImageStream.)
oc set triggers deploy/autoscale-arena-worker \
    --from-image=autoscale-arena-frontend:latest -c worker
oc set triggers deploy/autoscale-arena-frontend \
    --from-image=autoscale-arena-frontend:latest -c frontend

# Wait for it
oc rollout status deploy/autoscale-arena-worker
```

### Verify worker pod labels (required)

The SSE stream filters on `app=autoscale-arena-worker`. If the pod template is
missing that label, the HPA still works but the UI will silently show "no
pods" during the demo. Check before creating the HPA:

```bash
oc get pods --show-labels -l app=autoscale-arena-worker
```

- At least one pod listed → proceed to the HPA step.
- Empty output → apply the label patch from the Troubleshooting section
  below, then re-run this verification command before moving on.

## Apply the HPA

```bash
oc apply -f hpa.yaml
```

## Get the public URL

```bash
oc get route autoscale-arena-frontend \
    -o jsonpath='https://{.spec.host}{"\n"}'
```

Open it on your phone.

## Watch it happen

Run these in separate terminal panes before the demo:

```bash
# Pod scaling
oc get pods -w -l app=autoscale-arena-worker

# HPA state (CPU%/target / current / desired)
oc get hpa autoscale-arena-worker -w

# Worker pod logs (cycle through if many pods)
oc logs -l app=autoscale-arena-worker -f --max-log-requests=10
```

## Rebuild after code changes

```bash
git push origin main
oc start-build autoscale-arena-frontend --wait
# ImageChange triggers roll both deployments automatically
```

## Troubleshooting

- **`oc get hpa` shows `<unknown>/50%` for more than a couple of minutes.**
  Metrics-server isn't reading the pod. Double-check that the worker
  Deployment sets `resources.requests.cpu`. The HPA needs requests to
  compute a utilisation ratio.

- **Pods have no `app=autoscale-arena-worker` label.**
  `oc new-app --image-stream` can create a pod template without the label
  we filter on. Verify with `oc get pods --show-labels`. If the label is
  missing, either re-apply `worker.yaml` or patch the template:
  ```bash
  oc patch deploy/autoscale-arena-worker --type=strategic \
      -p '{"spec":{"template":{"metadata":{"labels":{"app":"autoscale-arena-worker"}}}}}'
  ```

- **Route returns 504 / SSE disconnects.**
  The Route already sets `haproxy.router.openshift.io/timeout: 4h`, and the
  frontend emits a 15-second SSE heartbeat. If the connection still drops,
  check the Route annotation is intact:
  ```bash
  oc get route autoscale-arena-frontend -o yaml | findstr timeout
  ```

- **Sandbox idles out.** The Developer Sandbox kills pods after ~12 hours
  of inactivity. Before a demo:
  ```bash
  oc rollout restart deploy/autoscale-arena-frontend deploy/autoscale-arena-worker
  ```

## Teardown

```bash
oc delete -f hpa.yaml -f worker.yaml -f frontend.yaml -f rbac.yaml --ignore-not-found
oc delete bc/autoscale-arena-frontend is/autoscale-arena-frontend --ignore-not-found
```
